# 2. Tunneling Guide ( Windows )

This guide is for exposing your local `n8n` beyond `localhost` when something outside your PC must call it.

Use this for:

- Stripe webhooks
- Telegram bot webhooks
- GitHub webhooks
- Typeform callbacks
- Meta callbacks
- any other public webhook or callback into your local `n8n`

You do **not** need this just for Codex talking to your local `n8n_live`.

## 1. Before You Start

Make sure all of these are already true:

1. your local `n8n` is already running
2. you can open `http://localhost:5678`
3. you understand that a tunnel exposes your local `n8n` to the internet
4. you are only doing this because an external service needs to call your local instance

Quick check in PowerShell:

```powershell
docker ps
```

If `n8n` is not running yet, go back to the local setup guide first:

- [1. Local Setup](./1.%20local%20setup.md)

## 2. Pick The Right Tunnel Type

### Option A: Cloudflare Quick Tunnel

Use this when:

- you want the fastest setup
- you only need temporary local webhook testing
- you are okay with a random temporary URL

Do **not** treat this as your final production setup.

### Option B: Cloudflare Named Tunnel

Use this when:

- you want a stable hostname
- you want something closer to a real long-running setup
- you have a domain on Cloudflare

This is the better Cloudflare option for long-running use.

### Option C: ngrok

Use this when:

- you prefer ngrok
- you want its traffic inspection and developer workflow

The free plan gives you one assigned dev domain.
If you want your own custom static domain, use a paid plan.

## 3. Cloudflare Quick Tunnel ( Step By Step )

This is the easiest path for most people.

### Step 1. Download `cloudflared`

Download the Windows x64 binary here:

- [https://developers.cloudflare.com/cloudflare-one/connections/connect-networks/downloads/](https://developers.cloudflare.com/cloudflare-one/connections/connect-networks/downloads/)

Recommended setup:

1. rename the file to `cloudflared.exe`
2. create this folder if needed:

```text
C:\cloudflared
```

3. move `cloudflared.exe` into that folder

### Step 2. Open PowerShell And Verify It Works

Run:

```powershell
cd C:\cloudflared
.\cloudflared.exe --version
```

If Windows says the command is not found, you are either in the wrong folder or the file is not named `cloudflared.exe`.

### Step 3. If An Empty Config File Was Created Earlier, Remove It

If you previously double-clicked `cloudflared` and it created an empty config file, quick tunnels can behave weirdly.

Check this folder:

```text
%USERPROFILE%\.cloudflared
```

If you see an empty `config.yml` or `config.yaml`, either delete it or rename it to something like:

```text
config.yml.bak
```

### Step 4. Start The Tunnel

Run:

```powershell
cd C:\cloudflared
.\cloudflared.exe tunnel --url http://localhost:5678
```

You should get a public URL like:

```text
https://something.trycloudflare.com
```

Copy that URL.
You will need it in the next step.

### Step 5. Restart `n8n` So Webhooks Use The Public URL

Stop and recreate your local `n8n` container with `WEBHOOK_URL` and `N8N_PROXY_HOPS=1` set.

Replace `https://your-public-url/` below with your real tunnel URL:

```powershell
docker stop n8n
docker rm n8n

docker run -d --name n8n -p 5678:5678 `
  -e GENERIC_TIMEZONE="Asia/Singapore" `
  -e TZ="Asia/Singapore" `
  -e N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=true `
  -e N8N_RUNNERS_ENABLED=true `
  -e WEBHOOK_URL="https://your-public-url/" `
  -e N8N_PROXY_HOPS=1 `
  -v n8n_data:/home/node/.n8n `
  docker.n8n.io/n8nio/n8n:stable
```

### Step 6. Re-Open `n8n`

Open:

- `http://localhost:5678`

Then check the webhook URLs shown inside your workflows.
They should now use your public tunnel URL instead of `localhost`.

### Step 7. Update The External Service

Now go to the service that needs to call your webhook and update the callback URL there.

Examples:

- Stripe webhook endpoint
- Telegram bot webhook
- GitHub webhook
- Meta callback URL
- Typeform webhook URL

If the service already had the old URL saved, replace it with the new tunnel URL.

### Step 8. Keep The Tunnel Running

Important:

- the PowerShell window running `cloudflared` must stay open
- if that window closes, the tunnel dies
- if the tunnel dies, outside services can no longer reach your local `n8n`

### Step 9. If The Quick Tunnel URL Changes, Repeat The Update

Quick tunnels are temporary.
If you restart the tunnel later, you may get a different public URL.

When that happens, repeat:

1. copy the new URL
2. restart `n8n` with the new `WEBHOOK_URL`
3. update the external service again

## 4. Cloudflare Named Tunnel ( Better / More Stable )

Use this when you want something closer to a real long-running setup instead of a random testing URL.

### What You Need

- a domain managed by Cloudflare
- a Cloudflare account
- your local `n8n` already running on `http://localhost:5678`

### Step 1. Create A Tunnel In Cloudflare

In Cloudflare Zero Trust / Tunnels:

1. create a new tunnel
2. add a public hostname, for example:

```text
n8n.yourdomain.com
```

3. point it to your local service:

```text
http://localhost:5678
```

4. copy the tunnel token

### Step 2. Install The Tunnel As A Windows Service

In PowerShell, from the folder containing `cloudflared.exe`, run:

```powershell
.\cloudflared.exe service install <YOUR_TUNNEL_TOKEN>
```

Replace `<YOUR_TUNNEL_TOKEN>` with the real token from Cloudflare.

This is the cleaner option if you want the tunnel to keep running without keeping a PowerShell window open.

### Step 3. Restart `n8n` With Your Real Public Hostname

Use your real hostname as the webhook URL:

```powershell
docker stop n8n
docker rm n8n

docker run -d --name n8n -p 5678:5678 `
  -e GENERIC_TIMEZONE="Asia/Singapore" `
  -e TZ="Asia/Singapore" `
  -e N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=true `
  -e N8N_RUNNERS_ENABLED=true `
  -e WEBHOOK_URL="https://n8n.yourdomain.com/" `
  -e N8N_PROXY_HOPS=1 `
  -v n8n_data:/home/node/.n8n `
  docker.n8n.io/n8nio/n8n:stable
```

### Step 4. Update The External Service Once

Now register:

```text
https://n8n.yourdomain.com/...
```

as the webhook / callback URL in Stripe, Telegram, GitHub, Meta, or whatever service you are using.

Because the hostname is stable, you should not need to keep changing it like a quick tunnel.

## 5. ngrok ( Step By Step Alternative )

If you prefer ngrok, this is the basic setup.

### Step 1. Download ngrok

Download it here:

- [https://ngrok.com/downloads/windows](https://ngrok.com/downloads/windows)

Extract it somewhere easy, for example:

```text
C:\ngrok
```

### Step 2. Add Your ngrok Authtoken

Create an ngrok account, copy your authtoken from the ngrok dashboard, then run:

```powershell
cd C:\ngrok
.\ngrok.exe config add-authtoken <YOUR_AUTHTOKEN>
```

### Step 3. Start The Tunnel

Run:

```powershell
cd C:\ngrok
.\ngrok.exe http 5678
```

You should get a public HTTPS URL.

Copy that URL.

### Step 4. Restart `n8n` With The ngrok URL

Use the same Docker restart pattern:

```powershell
docker stop n8n
docker rm n8n

docker run -d --name n8n -p 5678:5678 `
  -e GENERIC_TIMEZONE="Asia/Singapore" `
  -e TZ="Asia/Singapore" `
  -e N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=true `
  -e N8N_RUNNERS_ENABLED=true `
  -e WEBHOOK_URL="https://your-ngrok-url/" `
  -e N8N_PROXY_HOPS=1 `
  -v n8n_data:/home/node/.n8n `
  docker.n8n.io/n8nio/n8n:stable
```

### Step 5. Update The External Service

Paste the new ngrok URL into the third-party service that needs to call your webhook.

### Step 6. Understand The Free Plan Limitation

The free plan gives you one assigned dev domain.
If you want your own fixed custom domain, use a paid plan.

## 6. What Users Actually Need To Do

This is the short version:

1. run local `n8n`
2. start a tunnel
3. copy the public URL
4. restart `n8n` with `WEBHOOK_URL` set to that URL
5. set `N8N_PROXY_HOPS=1`
6. update the webhook / callback URL in the external service
7. keep the tunnel alive while testing

That is the whole job.

## 7. Important Notes

### You Do Not Need A Tunnel For Everything

You do **not** need a tunnel for:

- Codex talking to your local `n8n_live`
- local-only workflow building
- local node testing that does not require an outside callback

You only need it when something outside your PC must call your local instance.

### Quick Tunnels Are For Testing

Cloudflare quick tunnels and random ngrok URLs are fine for testing.
They are not the right final shape for a stable always-on production setup.

If you want a proper stable public URL, use a named tunnel or move to a real hosted setup later.

### A Tunnel Exposes Your Local Instance

Be careful.
A tunnel is not magic.
It is exposing your local `n8n` to the internet.

Do not treat a public local tunnel as a permanent secure production deployment just because it works.

## 8. Fast Troubleshooting

### `cloudflared` opens but does nothing useful

You probably just launched the binary directly.
Use PowerShell and run the command yourself.

### `cloudflared` quick tunnel is not working

Check for an empty config file in:

```text
%USERPROFILE%\.cloudflared
```

Rename or remove `config.yml` / `config.yaml` if needed, then run the quick tunnel command again.

### Webhook URLs still show `localhost`

You forgot to restart `n8n` with:

- `WEBHOOK_URL`
- `N8N_PROXY_HOPS=1`

### The outside service still cannot reach your webhook

Check all of these:

1. the tunnel is still running
2. the public URL is correct
3. the external service was updated
4. the webhook path in `n8n` is correct
5. your workflow is active if the service expects a production webhook

### You restarted the tunnel and now it broke

That usually means the public URL changed.
Update `WEBHOOK_URL`, restart `n8n`, and update the third-party service again.
