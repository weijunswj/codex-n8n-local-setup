# 3. Tunneling Guide ( Local Development )

This guide is only for **local development**.

Use tunneling when:

- n8n is running on your laptop or desktop.
- You need a public URL for webhooks, OAuth callbacks, or testing external triggers.

Do **not** use this guide for a VPS-hosted n8n instance.

If n8n already runs on a server with a real domain, go to:

- [4. VPS Hosting](./4.%20vps%20hosting.md)

## 1. Production Warning

n8n's own tunnel tooling is explicitly for **local development and testing** and is **not safe for production**.

Cloudflare Quick Tunnels are also for testing only.

If you want a stable always-on public setup, do not keep stretching a tunnel into a fake production stack. Use a VPS instead.

## 2. The Short Answer

- Use **ngrok** if you want the smoothest debugging experience.
- Use **Cloudflare Quick Tunnel** if you want the fastest free throwaway tunnel.
- Use a **VPS instead of a tunnel** if the instance is supposed to stay online all the time.

For most people building locally with webhook-heavy workflows, **ngrok is the best default tunnel** because the traffic inspection and request replay tools make debugging much less painful.

## 3. What You Must Set In n8n

When n8n is behind any reverse proxy or tunnel, set:

```powershell
WEBHOOK_URL=https://your-public-url/
N8N_PROXY_HOPS=1
```

That makes n8n generate the correct webhook URLs instead of using `localhost:5678`.

## 4. Option A - ngrok

### Why use it

ngrok is the best local-dev pick when you care about:

- inspecting incoming webhook payloads,
- replaying failed requests,
- seeing exactly what the upstream service sent.

### Start the tunnel

If `ngrok` is in your PATH:

```powershell
ngrok http 5678
```

If you are running it from a local folder:

```powershell
cd C:\ngrok
.\ngrok.exe http 5678
```

Then copy the HTTPS forwarding URL and set it as `WEBHOOK_URL`.

### Good fit

- Local webhook testing.
- OAuth callback testing.
- Debugging third-party integrations.

### Free-plan tradeoffs

The free plan has usage caps and a single assigned dev domain, so this is not the right shape for a permanent hosted n8n.

## 5. Option B - Cloudflare Quick Tunnel

### Why use it

This is the fastest free path if you just need a public URL quickly.

### Start the tunnel

If `cloudflared` is in your PATH:

```powershell
cloudflared tunnel --url http://localhost:5678
```

If you are running it from a local folder:

```powershell
cd C:\cloudflared
.\cloudflared.exe tunnel --url http://localhost:5678
```

Cloudflare will generate a random `trycloudflare.com` URL.

### Good fit

- Fast short-lived testing.
- Throwaway demos.
- Quick webhook experiments.

### Quick Tunnel limits

Cloudflare documents Quick Tunnels as testing-only and notes these limits:

- Random subdomain.
- 200 concurrent request limit.
- No Server-Sent Events support.

## 6. Recommended Workflow For This Repo

If [1. Local Setup](./1.%20local%20setup.md) is already done, use this order:

1. Start n8n locally.
2. Start the tunnel.
3. Set `WEBHOOK_URL` and `N8N_PROXY_HOPS=1`.
4. Restart n8n if required.
5. Re-register or re-test the webhook.

## 7. Troubleshooting

### Webhook Still Shows localhost

Make sure:

- `WEBHOOK_URL` is set.
- n8n restarted after the change.
- The value ends with a trailing slash.

### Cloudflare Quick Tunnel Breaks Under Real Usage

That is expected. Quick Tunnels are for testing only.

Fixes:

- Keep them for throwaway testing.
- Move to a VPS for anything long-lived.

### Tunnel URL Changed And Integrations Broke

That is normal with throwaway tunnel URLs.

Fixes:

- Update the external service callback URL.
- Restart n8n with the new `WEBHOOK_URL`.
- Move to VPS hosting for anything permanent.

### The Outside Service Still Cannot Reach Your Webhook

Check all of these:

1. The tunnel is still running.
2. The public URL is correct.
3. The external service was updated.
4. The webhook path in n8n is correct.
5. Your workflow is active if the service expects a production webhook.

## 8. Scope Note On Purpose

This document intentionally does **not** cover:

- Hostinger.
- VPS hosting.
- Permanent public hosting.
- Domain and HTTPS setup for a real server.
- Reverse-proxy deep dives for production.

Those belong in:

- [4. VPS Hosting](./4.%20vps%20hosting.md)

If you are done with local testing and want a real hosted n8n, stop using tunnels as a crutch and move to the VPS guide.
