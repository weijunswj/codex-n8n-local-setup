# 4. VPS Hosting for n8n

This is the proper guide for running n8n on a VPS with a real domain and a stable public URL.

Use this document if you want:

- A public n8n instance that stays online.
- Stable webhook URLs.
- A custom domain.
- Cleaner upgrades, backups, and scaling.
- A setup that is less fragile than tunneling.

## 1. The Short Answer

If your goal is **the cheapest realistic cost with the most convenience**, **Hostinger KVM 2** is the best default pick for most people.

Why KVM 2 instead of KVM 1?

- KVM 1 is cheaper and may be fine for a very light single-user setup.
- KVM 2 is still inexpensive, but the extra CPU and RAM give you more headroom for heavier nodes, AI calls, overlapping executions, and future queue mode.
- The convenience premium over the absolute cheapest raw VPS options is small enough that it is usually worth paying.

If you care mostly about **raw lowest monthly infrastructure cost**, **Hetzner** is cheaper.

If you care more about **generic cloud familiarity** than value, **DigitalOcean** is fine but not the best-value default for n8n at the low end.

## 2. Hostinger vs Caddy: Not The Same Thing

Do not think of this as **Hostinger vs Caddy**.

They do different jobs:

| Thing | What it is | Job in an n8n setup |
| --- | --- | --- |
| Hostinger | VPS provider | Gives you the server, storage, networking, and Hostinger-specific deployment tools. |
| Caddy | Reverse proxy | Exposes n8n on your domain and handles HTTPS. |
| Traefik | Reverse proxy | Same class of tool as Caddy. Hostinger uses this automatically for some new n8n deployments. |
| Tunnel | Ingress service | Exposes a private or local n8n instance through a third-party network. |

On new Hostinger VPS Docker Catalog installs for n8n, Hostinger says HTTPS is already configured automatically through Traefik, so Caddy is not required at the start.

On a generic VPS, a reverse proxy such as Caddy or Traefik is the normal pattern.

## 3. Why VPS Hosting Is Better Than Tunneling For Real Use

A VPS is better for always-on n8n because it gives you:

- A stable public domain instead of a throwaway tunnel URL.
- Fewer tunnel-specific limits.
- Cleaner TLS and DNS.
- Easier backups and updates.
- A proper separation between development and production.

Tunnels are great for testing. They are not the best foundation for a permanent automation system.

## 5. Recommendation By Scenario

### Default pick

**Pick Hostinger KVM 2** if you want the best balance of price, ease, and not regretting your life later.

### Budget fallback

**Pick Hostinger KVM 1** if you want to spend the least on Hostinger and expect light usage.

### Raw-cheapest serious alternative

**Pick Hetzner CX23 + Primary IPv4** if you care more about lowest ongoing infrastructure cost than convenience.

### When DigitalOcean makes sense

Pick DigitalOcean only if you already use it, want that ecosystem specifically, or value platform familiarity over price-performance.

## 6. Deployment Tracks

You have two sensible ways to host n8n on a VPS.

### Track A - Hostinger Fast Path

This is the best route if you want the least friction.

#### Option 1 - Hostinger n8n VPS template

Deploy:

- Ubuntu 24.04 with n8n.
- Ubuntu 24.04 with n8n ( queue mode ).

Hostinger says n8n comes preinstalled in Docker.

#### Option 2 - Hostinger Docker Catalog

Deploy the n8n app from Hostinger Docker Manager.

For supported apps such as n8n on a new VPS, Hostinger says HTTPS is automatically configured through the built-in Traefik reverse proxy.

#### Using your own domain

1. Point an A record for your chosen domain or subdomain to the VPS IP.
2. Open a terminal on the VPS.
3. For current Hostinger Docker Catalog installs, go to:

```bash
cd /docker/n8n
nano .env
```

4. Replace the default hostname with your own domain.
5. Restart the stack:

```bash
docker compose down
docker compose up -d
```

#### Updating later

You can update with Docker Manager or via SSH:

```bash
cd /docker/n8n
docker compose pull
docker compose down
docker compose up -d
```

#### Queue mode on Hostinger

If you deploy the queue mode template, Hostinger says it starts with 3 workers by default.

To scale workers:

```bash
docker compose up -d --scale n8n-worker=<number>
```

### Track B - Portable Standard Path On Any VPS

This works on Hostinger, Hetzner, DigitalOcean, or any normal Linux VPS.

Use:

- Ubuntu 24.04.
- Docker Compose.
- n8n.
- A reverse proxy such as Caddy or Traefik.

n8n's official VPS guides for DigitalOcean and Hetzner use Docker Compose plus Caddy.

Basic outline:

1. Create an Ubuntu VPS.
2. Install Docker and Docker Compose.
3. Point a subdomain such as `n8n.example.com` to the VPS IP.
4. Open ports 80 and 443.
5. Put Caddy or Traefik in front of n8n.
6. Set:

```bash
WEBHOOK_URL=https://n8n.example.com/
N8N_PROXY_HOPS=1
```

7. Start the stack.
8. Log in and complete onboarding.
9. Test editor access and an external webhook.

## 7. Reverse Proxy And HTTPS

n8n recommends using a reverse proxy in front of the app for TLS / SSL.

### On Hostinger

If you used the current Docker Catalog route on a new VPS, Hostinger already says Traefik handles HTTPS automatically for n8n.

That means you do not need Caddy on day one.

### On a generic VPS

Use Caddy or Traefik.

That is the clean normal pattern and keeps certificate renewal out of the n8n container itself.

### Direct TLS in n8n

This is possible through:

- `N8N_SSL_CERT`
- `N8N_SSL_KEY`

But then certificate renewal is on you, which is usually more annoying than using a reverse proxy.

## 8. Core n8n Settings You Should Not Skip

When n8n sits behind a reverse proxy, set:

```bash
WEBHOOK_URL=https://your-public-domain/
N8N_PROXY_HOPS=1
```

## 9. Backups And Updates

### Backups

If your VPS provider offers snapshots or backups, use them.

On Hostinger VPS, weekly backups are enabled by default.

### Updates

For Docker Compose based setups, the normal update flow is:

```bash
docker compose pull
docker compose down
docker compose up -d
```

Take a backup first.

## 10. Scaling And Queue Mode

Start simple unless you already know you need more.

- Personal or light usage: Hostinger KVM 1 can be enough.
- Safer default for real use: Hostinger KVM 2.
- Heavier parallel work or team use: move to queue mode and scale workers.

## 11. When A Tunnel Still Makes Sense

A tunnel still makes sense when:

- you are developing locally,
- you only need short-lived webhook testing,
- or you do not want a public server yet.

That belongs in:

- [3. Tunneling Guide](./3.%20tunneling%20guide.md)

For anything permanent, use a VPS.

## 12. Final Recommendation

If you want the lowest realistic cost without turning your setup into an annoying side quest, **Hostinger KVM 2** is the best default pick for this repo.

If you care only about raw monthly cost and do not mind more DIY work, **Hetzner CX23 + Primary IPv4** is cheaper.
