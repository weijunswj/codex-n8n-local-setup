# 4. VPS Hosting for n8n

This is the **beginner guide** for running n8n on a real VPS.

If you want the easiest path with the least nonsense, use **Hostinger**.

If your goal is **the lowest realistic cost with the most convenience**, the best default pick is:

- **Hostinger KVM 2**

Use this instead if you want the cheapest possible Hostinger plan and expect very light usage:

- **Hostinger KVM 1**

If you only care about raw monthly infrastructure cost and do not mind more DIY work, **Hetzner** is cheaper.

---

## 1. What this guide is for

Use this guide if you want:

- a public n8n that stays online,
- a real domain or subdomain,
- stable webhook URLs,
- less pain than tunnels,
- a setup you can update and back up properly.

If you are only testing locally, stop here and use:

- [3. Tunneling Guide](./3.%20tunneling%20guide.md)

---

## 2. The simple answer

If you are new to VPS hosting, do this:

1. Buy **Hostinger KVM 2**.
2. Deploy **n8n** using Hostinger's ready-made option.
3. Open the default n8n URL and create your admin account.
4. Point your own subdomain to the VPS.
5. Change the n8n domain in Hostinger.
6. Use that domain for your webhooks and callbacks.
7. Back up before updates.
8. Update with Docker Manager or `docker compose` later.

That is the normal path.

Do **not** overcomplicate this on day one.

## 6. Step-by-step: Hostinger setup for beginners

This is the part to follow if you just want it working.

### Step 1 - Buy the VPS

Buy:

- **Hostinger KVM 2**

If you really want to save money and expect light usage, buy:

- **Hostinger KVM 1**

### Step 2 - Deploy n8n

Use one of these:

#### Option A - n8n VPS template

Pick:

- **Ubuntu 24.04 with n8n**, or
- **Ubuntu 24.04 with n8n ( queue mode )**

This is the most obvious path if you want a prebuilt n8n box.

#### Option B - Docker Catalog n8n app

On a **new Hostinger VPS**, open:

1. **hPanel**
2. **VPS**
3. **Manage** on your server
4. **Docker Manager**
5. Deploy **n8n** from the catalog

For supported apps such as n8n on a new VPS, Hostinger says HTTPS is configured automatically through **Traefik**.

### Step 3 - Open n8n for the first time

After deployment, open:

```text
https://n8n.<your-vps-hostname>
```

Create your admin account.

Once you can log in, the basic install is working.

### Step 4 - Decide your real domain

Use a subdomain, not your main root domain.

Good example:

```text
n8n.example.com
```

This is cleaner than putting n8n on your main homepage domain.

### Step 5 - Point the domain to the VPS

Create an **A record** in your DNS that points your chosen subdomain to your VPS IP.

Example:

```text
Type: A
Name: n8n
Value: <your-vps-ip>
```

Wait for DNS to propagate.

### Step 6 - Change n8n to use your domain

For Hostinger's current n8n deployment flow:

1. Open **hPanel**.
2. Go to **VPS**.
3. Open your server.
4. Open **Browser terminal**.
5. Run:

```bash
cd /docker/n8n
nano .env
```

6. Replace the default Hostinger hostname with your own domain.
7. Save the file.
8. Restart n8n:

```bash
docker compose down
docker compose up -d
```

### Step 7 - Test the new domain

Open your domain in the browser:

```text
https://n8n.example.com
```

If it opens and you can log in, the domain part is done.

### Step 8 - Use this domain for integrations

When Stripe, Telegram, GitHub, Typeform, Meta, or OAuth providers ask for callback or webhook URLs, use your real n8n domain.

That is the whole point of moving to VPS hosting.

## 8. Updates: how to not break your setup

Before updating:

1. Take a backup or snapshot first.
2. Then update.

### Easy update path

Use Hostinger's Docker Manager / Docker Compose Manager and press **Update**.

### SSH / terminal update path

```bash
cd /docker/n8n
docker compose pull
docker compose down
docker compose up -d
```

After the update:

- open n8n,
- log in,
- trigger a small test workflow,
- check that webhooks still point to the correct public domain.

---

## 9. Backups: do this before you get clever

Hostinger VPS backups are **weekly by default**.

That is good, but still do this before big changes:

- take a manual snapshot if the option is available,
- or at least make sure you know which backup you would restore from.

Use backups before:

- upgrading n8n,
- changing domains,
- changing major environment settings,
- switching to queue mode.

---

## 10. Queue mode: ignore it until you actually need it

If you are new, start with normal n8n first.

Only care about queue mode when:

- you have heavier workloads,
- you run many executions at once,
- you want to scale workers separately.

If you deploy Hostinger's queue mode template, Hostinger says it starts with **3 workers** by default.

To scale workers:

```bash
docker compose up -d --scale n8n-worker=<number>
```

If that sentence looks confusing right now, good. Ignore it until you actually need it.

---

## 11. Important thing people confuse: Hostinger vs Caddy

These are **not** alternatives.

- **Hostinger** = the VPS provider.
- **Caddy** = a reverse proxy.
- **Traefik** = another reverse proxy.

On new Hostinger Docker Catalog n8n installs, Hostinger says **Traefik** already handles HTTPS automatically.

So for the easy Hostinger path, you do **not** need to bolt Caddy on top on day one.

---

## 12. What to do if webhook URLs look wrong

If you are using a **generic VPS / custom Docker Compose** setup instead of Hostinger's ready-made path, n8n says you should set:

```bash
WEBHOOK_URL=https://n8n.example.com/
N8N_PROXY_HOPS=1
```

That matters when n8n sits behind a reverse proxy and would otherwise try to generate URLs from its internal port instead of the public HTTPS domain.

For the simple Hostinger path, start with Hostinger's ready-made deployment first. Only go deeper into custom env tuning if something is actually wrong.
